# HDC Target UI — Diagnostics v2

Дата фиксации: 2026-05-25  
Статус: целевой вариант web-интерфейса HDC для следующей версии диагностического экрана.

---

## 1. Выбранный вариант

Целевым выбран вариант **“правый верхний”** из набора макетов HDC UI.

Краткое описание:

```text
Светлый интерфейс с боковой навигацией, карточками сервисов, верхним баннером общего состояния и компактными диагностическими блоками.
```

Этот вариант должен стать визуальной основой для `HDC Diagnostics v2`.

---

## 2. Общий стиль

Целевая визуальная модель:

```text
светлая тема по умолчанию;
чистый интерфейс в стиле Home Assistant;
много воздуха между блоками;
карточки с мягкими границами;
статусы через зелёные/жёлтые/красные бейджи;
минимум raw JSON на первом экране;
технические details скрываются в раскрываемые блоки.
```

Темная тема может быть добавлена позже, но целевой первый вариант — светлый.

---

## 3. Базовая структура экрана

Экран состоит из следующих областей:

```text
1. Верхняя шапка.
2. Боковая навигация.
3. Баннер общего состояния.
4. Сетка ключевых сервисов.
5. Блок сводки.
6. Блок свежести snapshot.
7. Блок последних блокировок действий.
8. Блок рекомендаций.
9. Таблица зависимостей и внешних сервисов.
```

---

## 4. Верхняя шапка

В шапке показывать:

```text
Home Dialog Console
Диагностика и мониторинг системы диалогов
```

Справа:

```text
Обновлено: HH:MM:SS
кнопка Обновить всё
```

Правила:

```text
время обновления брать из generated_at diagnostics;
кнопка обновления вызывает повторную загрузку данных;
автообновление можно оставить раз в 60 секунд;
формат времени должен быть человекочитаемый, не ISO-строка.
```

Пример:

```text
Обновлено: 16:32:14
```

---

## 5. Боковая навигация

Слева — постоянная навигация HDC.

Пункты:

```text
Обзор
Окружение
Qdrant / Source Selector
Промты
Анализаторы
Действия FastPath
Объекты и алиасы
База данных
Логи
Настройки
О системе
```

Для первого этапа активным должен быть только раздел `Обзор`. Остальные пункты могут быть видимыми, но вести на placeholder-страницы или быть добавлены постепенно.

---

## 6. Баннер общего состояния

Верхний баннер должен сразу отвечать на вопрос: “система работает или нет”.

Состояние OK:

```text
Система работает нормально
Все ключевые сервисы доступны и функционируют.
```

Состояние warning/degraded:

```text
Система работает с предупреждениями
Есть некритичные проблемы или устаревшие данные.
```

Состояние error:

```text
Система работает с ошибками
Одна или несколько ключевых зависимостей недоступны.
```

Справа в баннере — компактная карточка:

```text
Общее состояние
здорово / предупреждение / ошибка
```

---

## 7. Сетка ключевых сервисов

Главная сетка карточек должна показывать ключевые сервисы.

Минимальный набор:

```text
dialog-service;
Planner llama.cpp;
Redis;
Telegram runner;
HA API;
Source Selector;
Qdrant;
llm_log_helper;
Ollama fallback;
SQLite config DB;
Action Executor;
System Snapshot.
```

Карточка сервиса показывает:

```text
иконка;
название;
статус;
последняя проверка;
время ответа;
краткое описание роли сервиса.
```

Пример:

```text
Planner llama.cpp
Работает
Время ответа: 5 мс
Планирование задач и выбор действий.
```

## 9. Блок “Сводка”

Сводка должна показывать компактные числа:

```text
Всего проверок;
OK;
Ошибки;
Инциденты;
Рекомендации;
Блокировки действий;
Доступность.
```

Пример:

```text
Всего проверок: 13
OK: 13
Ошибки: 0
Инциденты: 0
Рекомендации: 1
Блокировки действий: 3
Доступность: 100%
```

---

## 10. Блок “Snapshot”

Показывать:

```text
статус snapshot;
возраст snapshot;
время последнего обновления;
краткое пояснение.
```

Пример:

```text
Snapshot
Статус: актуален
Возраст: 52 сек
Обновлён: 16:31:12
```

Правила:

```text
age_sec <= 180 → актуален;
age_sec > 180 → устарел;
файл не найден → ошибка.
```

---

## 11. Блок “Последние блокировки действий”

Источник:

```text
diagnostics.action_blocks
```

Блокировки действий не являются runtime-инцидентами.

Строка должна показывать:

```text
domain.service entity_id;
причина;
кто запросил;
время.
```

Пример:

```text
switch.turn_on switch.kettle_new_rk_g211s_power_2
action_not_in_positive_allowlist · telegram:... · 25.05.2026 14:20
```

Цвет блока — нейтральный/жёлтый, не красный.

---

## 12. Блок “Рекомендации”

Источник:

```text
diagnostics.recommendations
```

Если рекомендаций нет:

```text
Рекомендаций нет.
```

Если есть:

```text
заголовок;
текст рекомендации;
ссылка/кнопка на нужный раздел HDC, если применимо.
```

---

## 13. Таблица зависимостей и внешних сервисов

Внизу экрана — таблица:

```text
Сервис | Статус | Время ответа | Последняя ошибка | Рекомендация
```

Включать:

```text
Redis;
Home Assistant API;
Telegram runner;
Planner llama.cpp;
Source Selector;
Qdrant;
llm_log_helper;
Ollama fallback.
```

Для OK-состояния:

```text
Последняя ошибка: —
Рекомендация: Нет рекомендаций
```

---

## 14. Raw details

Raw JSON не должен занимать основное место на первом экране.

Правило:

```text
первый экран показывает человекочитаемые карточки;
подробный JSON доступен через раскрываемый блок “Подробности” или отдельную страницу Debug.
```

Для `Diagnostics v2` достаточно:

```text
<details>
  <summary>Подробности</summary>
  JSON
</details>
```

---

## 15. Поведение на мобильном

На мобильном:

```text
боковая навигация сворачивается;
сервисные карточки идут в одну колонку;
таблица зависимостей превращается в список;
кнопка “Обновить всё” остаётся доступной сверху;
raw details скрыты по умолчанию.
```

---

## 16. Источник данных

Первый экран HDC использует:

```text
GET /api/diagnostics/summary
```

HDC сам проксирует данные из:

```text
dialog-service /admin/diagnostics/summary
```

UI не должен напрямую обращаться к CT101, CT121, Qdrant или другим сервисам.

---

## 17. Версия реализации

Целевая версия для реализации этого UI:

```text
HDC 0.1.8 — Diagnostics v2 visual layout
```

Backend можно оставить:

```text
dialog-service 0.7.4
```

Если в процессе реализации потребуется добавить новые поля для UI, backend поднять до:

```text
dialog-service 0.7.5
```

---

## 18. Что не входит в этот экран

В Diagnostics v2 не входят формы редактирования:

```text
редактирование переменных окружения;
редактирование промтов;
редактирование анализаторов;
редактирование allowed_actions;
редактирование object_aliases.
```

Эти функции относятся к следующим экранам HDC.

---

## 19. Связанные будущие экраны

После Diagnostics v2 нужны отдельные разделы:

```text
Окружение — переменные окружения read-only;
Qdrant / Source Selector — статус, индексация, тест фразы;
Промты — просмотр prompt-файлов;
Анализаторы — список и настройки анализаторов;
Действия FastPath — allowed_actions и high_risk_actions;
Объекты и алиасы — rooms/entities/aliases/object_aliases;
База данных — таблицы SQLite config DB;
Логи — llm_log_helper и диагностические выдержки.
```

---

## 20. Критерии готовности

Diagnostics v2 считается готовым, если:

```text
1. Визуально соответствует выбранному правому верхнему макету.
2. Светлая тема выглядит как нативная панель Home Assistant.
3. Есть боковая навигация.
4. Есть верхний баннер общего состояния.
5. Есть сетка ключевых сервисов.
6. Есть сводка.
7. Есть snapshot-блок.
8. Есть блок последних блокировок действий.
9. Есть блок рекомендаций.
10. Есть таблица зависимостей.
11. Raw JSON скрыт по умолчанию.
12. Экран нормально работает на мобильном.
```
